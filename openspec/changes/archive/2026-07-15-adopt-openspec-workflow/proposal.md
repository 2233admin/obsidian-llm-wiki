## Why

LLMwiki now has several active engineering streams: release/onboarding convergence, Canvas and archify integration, multi-agent protocol work, and ongoing repo cleanup. These changes need a lightweight source of intent that survives chat context without replacing GSD, work-OS issue notes, or reviewed vault memory.

OpenSpec gives the project a repo-local proposal/spec/design/tasks loop for larger engineering changes so implementation can be reviewed against explicit requirements before code is touched and archived back into current specs afterward.

## What Changes

- Add OpenSpec as the preferred planning surface for non-trivial LLMwiki engineering changes.
- Keep OpenSpec scoped to repo-local development intent and change review; it does not own vault memory, MCP runtime behavior, GSD execution state, or work-OS issue state.
- Add a project-specific OpenSpec context and artifact rules that encode LLMwiki naming, source-registration, memory-governance, and work-OS boundaries.
- Add local Codex OpenSpec skills generated by `openspec init . --tools codex`.
- Document how to run the OpenSpec loop for this repository.

## Non-Goals

- Do not migrate existing GSD tasks, work-OS issues, or vault notes into OpenSpec.
- Do not require OpenSpec for tiny typo fixes, emergency hotfixes, or purely mechanical formatting.
- Do not introduce a new runtime dependency into the MCP server, compiler, fleet modules, setup scripts, or release artifacts.
- Do not register local repository paths as LLMwiki Sources; OpenSpec changes are repository planning artifacts, not Source Registry entries.

## Capabilities

### New Capabilities

- `openspec-governance`: Defines how LLMwiki uses OpenSpec proposals, specs, design docs, tasks, validation, and archives for engineering changes.

### Modified Capabilities

- None.

## Impact

- Adds `openspec/config.yaml`, `openspec/changes/adopt-openspec-workflow/`, and local `.codex/skills/openspec-*` command adapters.
- Adds repository documentation for using OpenSpec with LLMwiki.
- Does not change MCP tool schemas, Python/TypeScript runtime behavior, release packaging contents, or vault file layouts.
