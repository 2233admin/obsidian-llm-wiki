# Gate auto-promotion by Run Output Class, defaulting to human review

The Work Driver may close LLM Wiki's loop autonomously but must never become a source of knowledge drift, so it returns a Work Run's result into the vault only by Run Output Class under a fail-safe Promotion Policy. View outputs are compiled, never promoted (§0 invariant #2). Work-state transitions — an executable item moving through its workflow `state` — may auto-promote, because they are mechanical, git-reversible, and already guarded by the base-head optimistic lock (`HEAD_MISMATCH`) and the blocked-by / `effective_state` checks built in Task 8. Knowledge claims — any `fact`, `decision`, or `supersedes` about an entity — always route to triage for human review and never auto-promote, because auto-committing a wrong claim is exactly the drift LLM Wiki exists to prevent; this keeps §0 invariant #3 (promote only through the human PR gate) intact for the semantic layer. External side-effects (real git push, forge PR, thread reply) sit behind an explicit per-run approval that is stricter than promotion.

Auto-promotion is an allowlist a result must affirmatively clear, not a denylist — any output that cannot be classified as drift-safe falls back to human review. The trade-off accepted: less full autonomy in exchange for a non-negotiable guarantee that current-truth is never machine-authored.
